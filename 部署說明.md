# WU_客戶用說明網站 — GitHub + Vercel 快速部署說明

> 本文件給接手此專案的其他 Agent 閱讀，說明目前部署環境的建置方式、現況與後續操作流程。

## 一、架構選擇

本專案採用「純靜態網頁 + Vercel 自動部署」架構，**不同於**使用者慣用的架構 A（GAS 全包）與架構 B（GitHub Pages + GAS 後端 API）：

- 前端：純靜態 HTML / CSS / JS
- 部署平台：Vercel
- 部署流程：本機 `git push` → GitHub repo → Vercel 偵測 push 自動重新部署
- 若後續此網站需要資料庫或後端邏輯（例如表單送出、權限判斷），需另外評估加入 GAS API 或其他後端服務，目前尚未建置

## 四、後續 Agent 的快速部署操作流程（重要：勿在使用者資料夾內直接跑 git 指令）

**已知限制**：使用者選取的資料夾（`/mnt/...` 掛載路徑，包含 `WU_客戶用說明網站` 與暫存的 `outputs` 目錄）底層檔案系統**不允許刪除檔案**，只能新增／覆寫。這會導致 git 在該路徑下操作時，內部的 lock 檔（`index.lock`、`HEAD.lock`、`config.lock` 等）用完後無法刪除，第二次執行 git 指令就會出現「Another git process seems to be running」的錯誤且無法自行修復。

**正確做法**：git 操作一律在 `/tmp` 底下的暫存複本進行，完成後再把最終檔案複製回使用者資料夾供其檢視。流程如下：

```bash
# 1. 在 /tmp 建立乾淨的工作複本（不要在 mnt 路徑下 clone/init）
rm -rf /tmp/work/repo
git clone https://[GitHub帳號]:[Personal Access Token]@github.com/luka569/wu-customer-guide.git /tmp/work/repo

# 2. 把使用者資料夾內最新的網頁內容複製進工作複本（專案檔案已集中放在 wu-customer-guide 子資料夾內）
cp -r "[WU_客戶用說明網站 資料夾路徑]/wu-customer-guide"/* /tmp/work/repo/

# 3. 在 /tmp/work/repo 內做 git 操作
cd /tmp/work/repo
git add -A
git commit -m "[說明本次修改內容]"
git push

# 4. （非必要）若有新產生的檔案，可用 Write/Edit 工具或 cp 覆寫回使用者資料夾，
#    讓使用者在 Finder 也能看到最新內容（此步驟只能新增/覆寫，不能刪除舊檔）
```

push 完成後 Vercel 會自動重新部署，**不需要**呼叫 Vercel CLI 或 API（因為在 sandbox 中不可用）。

Token 建議請使用者當下提供（本文件不記錄實際 token 值），或詢問使用者是否已將 token 存放於其他安全位置供沿用。

## 五、網路限制注意事項（重要）

此 sandbox 環境的網路白名單會封鎖以下網域，遇到相關錯誤時不需重試，直接採用替代方案：

| 被封鎖網域 | 影響 | 替代方案 |
|---|---|---|
| `api.github.com` | 無法用 GitHub API 建立 repo、查詢帳號資訊 | 改用 `git` 指令操作（`github.com` 網域本身可連線） |
| `api.vercel.com` | Vercel CLI 完全無法使用（login / link / deploy 皆會失敗） | 請使用者於自己瀏覽器手動 Import GitHub repo 到 Vercel 一次，之後靠 GitHub push 自動部署 |

## 六、安全性注意事項

- GitHub Personal Access Token（名稱：`claude-deploy`，到期日：`2026-08-03`）不會寫入使用者資料夾內的任何檔案；每次部署時於 `/tmp/work/repo` 的 git remote URL 中臨時使用，該工作複本用完即可捨棄
- Token 到期或權限不足時，git push 會回傳 403，需請使用者重新產生 token 或調整該 token 的 Repository access（需包含此 repo）與 Contents 權限（需為 Read and write）
- 前端 JS（`script.js`）不得寫入任何權限判斷或敏感資訊；如未來加入後端邏輯，一律放在後端處理
